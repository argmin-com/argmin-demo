# Feature-to-Demo Coverage Matrix

This matrix treats the local demo mocks as product contracts. The demo may run
without cloud credentials, but each mock must preserve the production behavior
that matters for a credible stakeholder walkthrough: schema shape, lifecycle
status, failure mode, permissions posture, pagination semantics, deterministic
state, and explicit caveats.

For the concrete source markers that distinguish demo-only fixtures, fake auth,
local adapters, frontend mocks, and presenter scripts from production
implementation, see [demo-only-boundaries.md](demo-only-boundaries.md).

## Contract Rules

- All demo records are synthetic and local-only.
- Local mocks must expose the same business states the production system needs:
  ready, degraded, simulated, failed, unknown, provisional, approved,
  implemented, and fail-open.
- API mocks must return stable schemas and production-like error statuses for
  invalid payloads, unknown resources, disabled runtime roles, missing auth,
  oversized requests, rate limits, and unsafe demo operations.
- List mocks must support bounded `limit` pagination. A limit of `0` returns an
  empty page; out-of-range limits are rejected before handler execution.
- Any outbound integration remains simulated unless an operator explicitly
  enables live network delivery and holds the required notification scope.

## Current Gaps Kept Explicit

| Gap | Why it is acceptable for local demo | Required production path before customer deployment |
|---|---|---|
| Real cloud ingestion credentials are not used | The demo must be safe on a laptop and presentation-safe | Connector credential onboarding, source-specific backfills, and least-privilege deployment reviews tracked in [#1](https://github.com/argmin-com/argmin-demo/issues/1) |
| Notification delivery is simulated by default | Prevents accidental writes to Slack, email, Jira, ServiceNow, or webhooks | Customer-approved connector configuration, live scope, target allowlists, delivery retries, and audit export tracked in [#4](https://github.com/argmin-com/argmin-demo/issues/4) |
| Durable stores are optional | Memory mode gives the fastest deterministic demo path | Redis/Kafka/Neo4j, migration checks, retention policy, replay testing, and backup/restore validation tracked in [#3](https://github.com/argmin-com/argmin-demo/issues/3) |
| UI dataset is richer than some runtime mocks | Stakeholder walkthrough needs broad product story without production data | Runtime-backed read models for every represented surface and contract tests for each public API tracked in [#5](https://github.com/argmin-com/argmin-demo/issues/5) |
| No real tenant IAM or SSO lifecycle | Local demo uses synthetic accounts and bypass only in demo mode | Customer IdP integration, tenant-scoped RBAC, SCIM sync, break-glass policy, and audit retention tracked in [#2](https://github.com/argmin-com/argmin-demo/issues/2) |
