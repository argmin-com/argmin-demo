The interceptor sits in the critical path of the customer's production AI applications. The fail-open guarantee dictates
that under no internal failure condition will the platform drop, delay, or mutate a legitimate client request. This matrix
defines the mandatory chaos engineering tests for the Phase 0 Glass Jaw validation suite (PRD NFR-11).
Principle: if the system cannot evaluate a request safely, it MUST pass the request through unmodified. Every
fail-open event is logged asynchronously for post-hoc reconciliation.

1.4 Monitoring Requirements
The platform must emit the following metrics for fail-open observability: fail_open_total (counter by reason),
fail_open_rate (gauge, 1m/5m/15m windows), interceptor_latency_p99 (histogram), circuit_breaker_state (gauge:
closed/open/half-open). Alert threshold: fail_open_rate > 1% over any 10-minute window triggers PagerDuty/Slack
notification.

2. Active-Lite Specification
Active-Lite is a restricted enforcement mode that allows safe request shaping and model routing under a strict Do No
Harm mandate. It enables the platform to demonstrate real-time control to design partners (proving optimizer, not just
observability) using only mathematically and semantically safe operations.

2.1 Semantic Safety Bypass (Rule 1)
Before any payload modification, the interceptor parses the outbound JSON. If the payload contains any of the
following, ALL Active-Lite modifications are bypassed and the request downgrades to Advisory mode: (a)
response_format: { type: 'json_object' }, (b) strict JSON schema definitions, (c) tools or functions arrays. Rationale:
tool-calling schemas and structured generation capabilities vary between models; swapping or capping these
requests guarantees breakage.

2.2 Environment Gating (Rule 2)
Active-Lite modifying or blocking actions execute ONLY if request metadata resolves to environment: staging, dev, or
test. If environment resolves to production (or is null/unresolved), the interceptor defaults to Advisory mode.
Production opt-in requires explicit per-service configuration override.

2.4 Explicitly Forbidden Actions
• Request blocking in production without per-service opt-in
• Payload rewriting beyond max_tokens injection (and only where semantic safety bypass does not apply)
• Prompt content modification
• Response content modification or truncation
• Automatic service shutdown
• Rate limiting (handled by customer's own infrastructure)
• Cross-provider model routing (e.g., OpenAI to Anthropic; deferred to Phase 2)

2.5 Configuration and Rollback
Active-Lite must be explicitly enabled per service via configuration (default: disabled). Configuration specifies allowed
actions per service. Immediate disablement: setting active_lite_enabled=false propagates within 60 seconds. All
modifications include X-ACI-Intervention response header identifying the action taken. Every modification generates
an intervention_event in the append-only audit log.

2.6-2.10 Additional Active-Lite Specifications
MVP Provider Scope (2.6): OpenAI models and AWS Bedrock models in MVP. Azure OpenAI, Vertex AI, and
self-hosted models are Phase 2.
Environment Trust Model (2.7): X-ACI-Env request header is treated as a hint only. Canonical environment derived
from trusted deployment metadata (Kubernetes namespace labels, cluster identity, IAM role/account mapping). If
environment cannot be resolved, treated as production (most restrictive).
Cache Hard Caps (2.8): max_entry_bytes: default 2MB. max_cache_bytes_per_service: default 100MB.
max_cache_bytes_total: default 1GB. Cache write sampling rate: default disabled (100%).
Gate Error Response Schema (2.9): HTTP 413 (token_size_exceeded), HTTP 403 (model_not_allowed), HTTP
429 (budget_exceeded). All return well-formed JSON error bodies with request_id, policy_id, and actionable details
(approved_alternatives for 403, retry_after_seconds for 429).
Confidence Thresholds (2.10): Shape actions require minimum attribution confidence of 0.90. Gate actions require
0.95 or higher, or explicit admin allowlist waiver.

4.0 End-to-End System Architecture
The complete system architecture includes telemetry sources, ingestion and normalization layer, core platform
components, decision-time interception path, and the optional vendor control plane. The customer trust boundary
(VPC) is explicitly delineated. Hot path (interceptor) depends only on Attribution Index (Redis) and in-process policy
bundle evaluator. Cold path (ingestion/graph) is eventually consistent.

4.2 Optional Vendor Control Plane
A non-sensitive vendor control plane (hosted outside customer VPC) may optionally provide: license/feature flag
management, update registry (signed container images, SBOM), version/health metadata collection. No customer
data, no prompts, no attribution data, no PII transits to vendor. Outbound connection only. Airgapped deployment
available for regulated environments.

4.3 Network and IAM Requirements
All pods in private subnets. Egress allowlist: AWS APIs, Okta API, GitHub API. IAM: prefer IRSA. Separate roles for
collectors (read-only) and interceptor (bedrock:InvokeModel scoped to approved model ARNs only). PII
pseudonymized via HMAC-SHA256 with KMS-protected customer key. HMAC key rotation: annual minimum. On
rotation, previous key retained for 30-day transition; graph edges re-derived during transition window.

6. Attribution Index (Redis) Data Model
The Attribution Index is the hot-path serving cache used by the interceptor for O(1) lookups. Deployed as Redis (or
KeyDB) inside the customer VPC in the aci-core namespace. The graph and timeseries stores are the durable
sources of truth; the index is derived, rebuildable, and eventually consistent.

6.1 Keying Strategy
Key format: aci:idx:v1:{env}:{rf_hash}. rf_hash = base32(sha256(resource_fingerprint))[:26] (~130 bits).
resource_fingerprint = hash(X-ACI-Service-Id + X-ACI-Route + provider_endpoint + [X-ACI-Tenant-Id if multi-tenant]).
Time-bucketing excluded from hot-path key to prevent thundering-herd cache invalidation.

6.3-6.10 Index Operations
TTL Strategy (6.3): Every index key MUST have a TTL. Default: 30 minutes. Sliding TTL on write.
Versioning and Invalidation (6.4): schema_v (data format) and index_version (content changes). index_version
MUST be monotonically increasing. Updated atomically via Lua script.
Write Path (6.5): Concurrent writers use Lua script for atomic upsert: TIME, HGET index_version, increment, HSET
all fields, PEXPIRE.
Read Path (6.6): Two-tier: (1) local in-process LRU micro-cache (1-5s TTL), (2) on miss, pipelined HMGET + PTTL to
Redis with strict 15ms timeout. On miss/timeout, fail-open.
Memory Sizing (6.7): Redis maxmemory policy: allkeys-lru. ~10,000 keys = ~5MB. Well within 256MB budget.
Semantic cache MUST use separate Redis logical database or dedicated instance.
Failure Modes (6.8): Redis unavailable: fail-open. Stale entry: TTL bounds staleness. Key collision: negligible at 130
bits.
Observability (6.9): aci_idx_lookup_p50/p95/p99_ms, aci_idx_hit_rate (micro-cache), aci_idx_key_count,
aci_idx_evictions_total, aci_idx_entry_age_ms.

11. UI Component Specifications (v1.7)
This section defines implementation specifications for UI components introduced or modified in PRD v2.0. These
specifications complement the functional requirements and provide engineering guidance for the frontend.

11.1 Manual Attribution Mapping Interface
The Attribution page includes a manual mapping panel that surfaces unresolved or low-confidence attributions. For
each unresolved item, the interface displays: request or service identifier, the platform's best-guess attribution with
confidence interval, and fields allowing the user to select or type the correct association.

Supported mapping targets:
• Teams
• Services
• Cloud environments
• CI/CD pipelines
• Git repositories
• Other infrastructure components as configured

Behavior on confirmation:
When a user confirms a manual attribution: (1) a loading indicator appears (see Section 11.2), (2) the attribution is
recorded with source='manual' and confidence=1.0 (direct user confirmation), (3) all affected metrics recalculate:
attribution coverage %, AI spend by team, optimization potential, provider breakdown, cloud breakdown, (4) the
manual input is fed into the HRE calibration loop (FR-212) as a ground truth sample. (5) The request moves from
'unresolved' to 'confirmed' status in the mapping interface.

11.2 Dynamic Recalculation Indicator
When user actions trigger metric recalculation (manual attribution, intervention approval/rejection, policy change), the
system displays a visible loading indicator:
• Visual: a centered overlay spinner with the text 'Recalibrating...' displayed over the affected content area.
• Duration: 0.9 to 1.2 seconds (configurable). This is a deliberate UX decision to signal that computation is
occurring, not a performance limitation.
• Scope: the indicator covers the content area being recalculated. Global metrics in the header (if visible) also
refresh.
• Completion: on completion, all affected metrics display updated values. No page reload required.

11.3 Collapsible Request Stream Panel
A slide-out right panel (collapsible drawer) accessible from any page via a persistent icon in the top navigation bar.
This replaces the inline execution stream that was previously displayed on the Overview dashboard.
• Default state: collapsed (hidden).
• Trigger: persistent icon/button in the top navigation bar.
• Content: most recent N inference requests (configurable, default 50) with: request ID, model, service,
estimated cost, attribution confidence, attribution status (resolved/unresolved/manual).
• Layout: panel width approximately 30-40% of viewport. Does not push or disrupt the primary page content.
• Accessibility: available from every page in the application.

11.4 Scenario-Based Cost Forecasting
The cost forecasting view (PRD FR-402) supports intervention-based scenario modeling. Implementation
requirements:
• Scenario selection: users can toggle individual interventions on/off or use preset buttons: 'No interventions',
'All interventions'. The forecast chart updates to reflect the selected scenario.
• Time horizons: selectable from a dropdown or button group: 3, 6, 9, 12, 18, 24 months.
• Visualization: line chart showing projected spend over the selected horizon with confidence band. Current
spend trajectory (no interventions) shown as baseline. Selected scenario shown as overlay.
• Forecast model inputs: removed from the primary forecasting page. Available in a dedicated configuration or
advanced settings view accessible via a settings icon or link.

11.5 Mode Switching Interface
The deployment mode selector (Passive Observation / Advisory / Active-Lite) must be visible and accessible from
primary navigation. Implementation:
• Location: visible in the top navigation bar as a labeled pill or badge indicating the current mode.
• Interaction: clickable. Expands to show a brief description of the current mode, what the other available
modes are, and what each mode means for the customer's AI workloads.
• Mode descriptions: Passive Observation (log-only, no headers), Advisory (enriches response headers with
cost and attribution data, no modifications), Active-Lite (may route, cache, or gate requests per policy).
• Mode change: changing mode triggers the recalculation indicator (Section 11.2) and updates the Governance
view.

11.6 Technical Diagnostics Separation
Operational metrics intended for platform engineering users are housed in a dedicated Technical Diagnostics tab,
separate from the primary user-facing views. This tab includes: data freshness per data source (Section 7), ingestion
latency, telemetry health, collector status (connected/erroring), event bus throughput, index materialization lag, and
circuit breaker state. These metrics are not displayed on the Overview, Attribution, or any other primary view.

11.7 Intervention Card Methodology Indicators
Each intervention card in the Interventions view displays:
• Optimization category: the predefined class (model over-provisioning, duplicate requests, staging/dev
sprawl, orphaned resources, prompt inefficiency, batch mispricing).
• Triggering data threshold: e.g., "Triggered: 2.1M fallback requests detected, 97.3% achieve equivalent
quality with cheaper model."
• Rule/condition met: e.g., "Rule: Model cost differential >50% with verified quality equivalence."
• Visibility: this information is displayed prominently on the card, not hidden behind a 'Show Details' toggle.

11.8 Overview Clickable Navigation
Every metric card on the Overview page (total AI spend, attribution coverage, optimization potential, savings
captured) is clickable and navigates directly to the associated detail view. Clicking attribution coverage navigates to
the Attribution page. Clicking optimization potential navigates to the Interventions page. Clicking savings captured
navigates to a filtered view of implemented interventions.

11.9 Terminology Standardization

All user-facing instances of "waste" are replaced with "optimization potential." This applies to: variable names in demo

12. Known Codebase Issues (v1.7)
This section documents known deviations between the codebase
(github.com/argmin-com/argmin-demo) and the PRD/Engineering Specification. These must be
resolved before design partner deployment.

12.1 _handle_org_change: Old Team Membership Edge Not Closed
File: src/aci/core/processor.py, _handle_org_change method
Bug: When processing an organizational change event (person moves from Team A to Team B), the handler creates
a new edge from the person to the new team, but does not explicitly close the old edge from the person to the old
team. The GraphStore.add_edge method closes existing edges with the same (from_id, to_id, edge_type) key, but
since the new edge has a different to_id (new team), the old edge remains active. The person appears as a member
of both teams simultaneously until the old edge expires via TTL.
Impact: Incorrect dual-team attribution. Cost attributed to person may split across both teams. Attribution confidence
for the affected person's workloads will be inaccurately inflated.
Fix: Before creating the new edge, explicitly close the old edge by calling graph.add_edge with the old team's to_id,
which will trigger the existing-edge closure logic. Alternatively, add a close_edge method to GraphStore that closes
an edge by (from_id, to_id, edge_type) key without creating a replacement.
Severity: High. First identified in v1.6 codebase audit.

12.2 TRAC Calculator: Carbon Liability in MVP Formula
File: src/aci/trac/calculator.py
Issue: The TRAC calculator computes Carbon Liability (emissions_kg_co2e * carbon_price_per_tco2e / 1000.0) and
includes it in the TRAC sum. PRD v2.0 (FR-704) specifies that the MVP formula is Billed Cost + Confidence Risk
Premium only, with Carbon Liability deferred to post-MVP.
Fix: Add a configuration flag (e.g., include_carbon_liability, default False for MVP) that skips the carbon liability
computation. When False, TRAC = billed_cost_usd + risk_premium.
Severity: Medium. Code vs. spec divergence. No runtime impact in MVP since carbon data is not ingested, but
creates a correctness gap if carbon emissions data is provided (e.g., from test fixtures).
